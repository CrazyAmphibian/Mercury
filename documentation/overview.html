Mercury

Mercury is an interpreted dynamically-typed language with a simple but robust syntax.

Why Mercury?

Mercury is here to show that a high-level language doesn't have to compromise on programmer freedom for the sake of simplicity. Drawing from the simplicity of C and ease of use from Lua, Mercury gives the user a lot of control and flexibility, while at the same time, keeping functions atomic and varied.

Mercury is also easy to embed into another program, and has been written in a way that makes it very accessible from the C++ foundation of the language.

Featuring things such as multithreadding, automatic garbage collection, and first class functions, Mercury is more than capable of standing up as a standalone language.

Design philosophy

Mercury is designed around a core set of ideas, listed in order of priority.

Mercury should be:

  • Fast

    • Mercury should be as fast as it can be, without sacrificing other capabilities

  • Simple

    • Mercury's syntax should be easy to read, and not tie multiple behaviors to a single syntax that would cause ambiguity for the user or compiler.

  • Flexible

    • Mercury should be ambivalent to any specific style of programming

    • Mercury should have a diverse set of standard library functions

  • Compatible

    • Mercury should be able to run on any system with the same (or similar) results

    • Mercury should not needlessly remove, deprecate, or change old behavior



Basic Syntax

Hello world

Hello world in Mercury is quite straightforward

print("Hello, world!")

Comments

Comments follow the same syntax as C.

A single line comment is done with two forward slashes

print("Hello, world") //Quite important, do not remove!

A multi-line comment is done as such

/*

this does some stuff.

Pretty cool stuff.

*/

print("Hello, world")

if two comment styles overlap, the multi-line takes priority.

// This does stuff /*

now it doesn't

*/

Order of operations



The program executes in the following order during runtime:

  1. Variable is retrieved

  2. Variable is indexed

  3. Function is called

  4. Unary operations are applied

  5. Binary operations are applied

  6. Variable is set

All operations of the same type are done in left-to-right order, and do not follow standard order of operations. The exception to this is unary operations, which are read from right to left.

For example:

b=#a+1/3

first, the variable 'a' is retrieved. Next, the length operator is applied to it. Then, it is added with 1, and divided by 3. Finally, the variable 'b' is set to the result

this example showcases how operations are applied in order

a=5+3/4 //returns 2.0

a=3/4+5 //returns 5.75

a=5+(3/4)//returns 5.75

Strings

Strings are constructed with two quote marks and work like typical strings. Strings will run until they find another quote mark, even if that means spanning multiple lines. Strings accept escapes using backslash, and follow similar rules to C string escapes.

Keywords

Mercury has 17 reserved keywords. These cannot be used for variable names, as they either are part of the language syntax, are represent another variable.

Keywords are case sensitive, so you can use capitalization to share a name with a keyword without any issues.

  • nil

  • true

  • then

  • else

  • goto

  • while

  • false

  • local

  • elseif

  • return

  • continue

  • function


keywords: Static variables

Static variable keywords represent a constant value, and can be used for assignment or comparison

  • nil

    • represents the nil datatype, nothing.

  • inf

    • represents infinity in as a float.

  • true

    • represents the boolean value of true

  • false

    • represents the boolean value of false

keywords: flow control

These keywords form the structure of a program and are vital for complex behavior

  • end

    • the universal delimiter for statement bodies.

  • if, then, else, elseif

    • forms the basis for if statements

  • while, do

    • forms the basis for while loops

  • continue, break

    • modifies behavior of loops

  • goto

    • unconditional jump to another section of code, use with caution

  • function

    • self-explanitory

  • return

    • can be used in function to abort execution and return values

keywords: scope specifiers

These keywords are used for variable retrieval and assignment to fine tune scope.

  • global

    • specifies that a variable should belong to the top-level state

  • local

    • specifies that a variable should belong to the bottom-most state

Operators

Operators: Arithmetic

Arithmetic operators work only on numbers (either floats or ints). In most cases, if an operator is provided with two ints, the result will always be an int, and if a float is provided, the result will be a float.

  • Addition +

  • Subtraction (and unary negation) -

  • Multiplication *

  • Division /

    • always returns a float

  • Exponentiation ^

    • always returns a float

  • Modulo %

    • operates on C modulo rules (truncation)

  • Integer Division \

    • always returns an int.

  • increment ++

    • only on environmental variables as a complete statement

  • decrement --

    • only on environmental variables as a complete statement


Operators: Bitwise

Bitwise work much like arithmetic operators, in being able to take both ints and floats. The returned type is always equal to the type of the first (left) operand. This can be used for casting floats to ints, or other niche tricks.

  • Bitwise not !

  • Bitwise and &

  • Bitwise or |

  • bitwise xor ~

  • bitshift left <<

  • bitshift right >>

Operators: Logical

Short-circuit evaluation is not provided. The returned value of logical operators is highly variable, dependent on each operator, but the boolean state will be maintained for the purpose of conditional checks.

  • Logical not !!

    • will always return true or false

  • Logical and &&

    • if true, returns the second value, otherwise returns false.

  • Logical or ||

    • if true, returns the first true value, otherwise returns false.

  • Logical xor ~~

    • if true, returns the value of the single true operand. If false, returns the value false.

Operators: Comparison

  • Greater than >

  • Less than <

  • Greater than or equal to >=

  • Less than or equal to <=

  • Equal ==

  • Not equal !=

Operators: listlike

  • Length #

    • strings and arrays will return their length (highest index). Tables will return the number of elements stored of all types.

  • Index [ ]

    • strings can be indexed as read only, attempting to assign to an index will result in an error.

Operators: Other

  • Concatenate ..

  • Call function ( )

Operators: Creation

  • New array [ ]

  • New table { }

  • New string " "



Control Flow

If statements

If statements in Mercury operate in a very similar way to those in other high level languages. They are split into 2 parts, the condition and the body. If the condition is true, then the contents of the body are executed. If statements start with if, followed by the condition. Next, the then keyword is used before the body, and is finished with end.

if a>20 then

print("a is kinda big")

end

the else keyword can be used to add more complexity to an if check, and is quite simple to use

if a>20 then

print("a is kinda big")

else

print("a is not that big")

end

the final layer is elseif, which combines else and another if statement together. Elseif can be used as many times as desired, but should go before else.

if a>20 then

print("a is kinda big")

elseif a<0 then

print("a is negative!")

else

print("a is not that big")

end

while loops

As with if statements, while loops are simple in operation

n=0

while n<10 do

do_stuff(n)

n++

end

the continue and break keyword can add further complexity.

n=0

while n<10 do

if should_abort(n) then

break

end

n++

if n%2==0 then

continue

end

stuff(n)

end

functions and returns

Functions are designated with the keyword and parameters. Do note that a function may receive less parameters than specified.

The return keyword will cause the immediate closure and escape from the current state. Values may be supplied to pass values to the calling state. Returning in the main body of the program will halt all execution.

Functions can either be assigned directly to a variable, or passed as an arg to another function.

add=function(num1,num2)

return num1+num2

end

Goto

Goto can jump code execution to any location within the current state. Caution is advised when using goto, as it can potentially jump over variable declarations and cause bugs with scoping. It's recommended to explicitly state global or local keywords to be safe.

a=0

b=0

while a<100 do

while b<100 do

if something_wrong(a,b) then goto exit end

other_stuff()

end

stuff()

end

exit:

for an example of incorrect usage of goto, this results in an error, since the label thing is not visible within the function state

stuff=function()

goto thing

end

thing:

stuff()



Structure semantics

Strings are created by enclosing characters in quote marks. Strings span across any amount of characters until another quote mark is hit, including newlines.

msg="hello

world!"

Strings are evaluated at tokenization, so longer strings do not add significantly more overhead, and escapes add no overhead.



Arrays and tables share the same creation syntax, albeit arrays use [ and ], while tables use { and }. The compiler will not check if the index is valid when evaluating creation. There are 2 syntax modes to arrays and tables. Implicit indexing, and explicit indexing.

my_array=[1,2,3] //implicit indexing. 0=1, 1=2, 2=3

my_array=[0=1,1=2,2=3] //explicit indexing. Results in the same code and data as above.

Explicit indexing can take named variables as well, which results in a get call.

a=5

my_array=[a=1] // 5=1

This should not be confused with an explicit string index

my_table={"a"=1} //doing this with an array would result in an error.



All structures can be indexed with brackets, with the variable you want in brackets

my_array=[1,2,3]

print(my_array[0]) // 1

Tables can be indexed with period notation, allowing for a cleaner way to index with a string

my_table={"a"=1}

print(my_table["a"]) // 1

print(my_table.a) // 1. not to be confused with my_table[a]



The Compiler

Mercury's compiler is (intentionally) very simple, generating code with two single-pass phases. The extent of the compiler's knowledge is at most backreferences to jump labels, and sometimes a couple of tokens ahead. This means that the compiler has very little ability to detect any code issues or inevitable runtime errors, requiring more careful programming.

Phase 0: source code

Consider the following program for the example:

print("Hello, world!")

a=5

if a>0 then

print("a is positive")

end

Phase 1: tokenization

The first phase of the compiler is to convert the source code into tokens to make the rest of the compilation easier and simpler. The tokenization phase cannot raise compiler errors, and will run from the first to the last character of the source code. Once the tokens are split from source code, some basic processing on a per-token basis is done to categorize them. Whitespace characters (including semicolon) are completely ignored when tokenizing (outside of separating symbols and strings)

Tokens themselves store the associated source code text, as well as position in the code, and the type of the token.
